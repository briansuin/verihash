# **🗺️ VeriHash Nexus v0.3-4 产品架构与开发路线图**

**产品定位**：AI 时代的工作量公证机，脑力劳动者的**记录、广播与可查询**身份基础设施 (Record, Broadcast, Query)。

**核心受众**：律师、研究员、独立开发者、法律极客、跨境合规专家。

**战略切入**：单机存证开局 \-\> 异构网络异步广播 \-\> Ask VeriHash 动态查询 \-\> 纯前端万能解析(默认官方可选网关/支持自主部署)。

**核心准则 (The Manifesto)**：

1. **隐私隔离**：AI 默认仅见修改片段，完整原文永不离机。  
2. **被遗忘权**：数据主权意味着“有权向世界广播”，更意味着“有权随时让世界闭嘴（逻辑撤销与物理抹除）”。  
3. **极简基建 (Fat Protocol, Thin App)**：专业平台做专业的事，绝不自建中心化数据库存储用户凭证，永远保持前端展示层的可选性与可替代性。

## **🧱 Phase 1：物理基建与时间感知层 (The Silent Sentinel & Time Anchor)**

**状态**：✅ 已完成

| 模块 | 机制 | 状态 |
| :---- | :---- | :---- |
| **身份引擎** | Ed25519 公私钥对，DID 本地生成，私钥永不离机。 | ✅ |
| **静默监听** | fsnotify 守护进程，监听绑定工作区目录。 | ✅ |
| **时间序列账本** | SQLite，记录 (时间戳, 文件增量, 当前哈希, 上一哈希)。 | ✅ |
| **时间轴感知** | **\[核心护城河\]** os.Stat 提取真实元数据。强制打入严谨时间戳：birth\_time (若系统不支持则标 unavailable), modification\_time, ingestion\_time。突破系统盲区。 | ✅ |

## **🧠 Phase 2：可插拔 AI 评估层 (The Pluggable Oracle)**

**状态**：✅ 已完成

* **云端默认 \+ 选择性披露**：主推大模型 API，外发脱敏截断片段。  
* **Mint 输入策略**：默认优先保证语义完整性。对于法律意见、合同定稿、一次性生成文档等高价值文件，优先采用全文或大段正文审计；对于局部修订、增量更新或超大文件，再选择性使用 Diff / 关键段提取以控制 token 与隐私边界。  
* **历史深度审计 (Historical Audit)**：AI 增加对历史时间轴的审查维度，结合 OS 时间戳，赋予旧档案“资历权重”。

## **📜 Phase 3：本地凭证与渲染层 (The Trojan Horse)**

**状态**：✅ 已完成

* **JSON 凭证生成**：生成标准化 JSON-LD。  
* **HTML 赛博战报导出**：渲染为单文件 HTML 卡片，便于 P2P 发送。

## **📤 Phase 4：异构广播与主权销毁层 (Heterogeneous Broadcast)**

**状态**：🔥 核心攻坚中 (MVP阶段)

*工程约束：采用统一适配器接口。异步广播层必须支持 Retry (重试), Backoff (退避), Idempotency (幂等)。以 vc\_id 为平台侧去重锚点，避免重复广播与状态漂移。网络失败绝不回滚本地账本。*

| 模块 | 战略定位 | 实现逻辑 | 状态 |
| :---- | :---- | :---- | :---- |
| **4.1 GitHub Gist** | **首个可验证广播通道** | 原子化模式 (One VC \= One Gist)。作为轻量级网关出口，而非唯一依赖。 | 🔄 开发中 |
| **4.2 Sovereign Revoke** | **被遗忘权的完美实现** | **逻辑撤销优先，物理删除辅助**：发布 Revoke 记录（Tombstone），并同步调用 API 物理抹除原 Gist。**Tombstone 需写入独立的公共根索引（Phase 4.5），绝不依附于被删除对象本身**。 | 🔄 开发中 |
| **4.5 Public Root Index** | **身份资产寻址之根** | **\[新增护城河\]** 发布最小化的 profile\_index.json。提前构建用户级索引，列出最近公开 VC、撤销状态(Revoked)、技能摘要。确保广播数据从第一天起就有根可寻。 | 🔄 开发中 |
| **4.3 Nostr 协议中继** | **Web3 抗审查广播** | 将签名 VC 作为 NIP-01 广播。支持通过 NIP-09 指令实现网络层删除。 | 📋 规划中 |
| **4.4 ❌ 废弃 IPFS** | **\[安全红线\]** | IPFS 永久存储特性与法律行业的撤回需求存在致命冲突，永久剔除。 | ❌ 废弃 |

## **🔍 Phase 5：可查询推理层 (Ask VeriHash \- The Query Layer)**

**状态**：🔥 核心战略新增 (A2A 的灵魂)

*战略定位：将死板的“凭证集合”转化为“可回答问题的能力接口”。*

* **Phase 5A (Local Query)**：本地 Ask VeriHash。基于本地 SQLite 配合规则筛选（时间/关键词），拼接 Prompt 调用大模型，先服务用户自身的资产检索。  
* **Phase 5B (Public Interface)**：对外可调用查询接口。  
  * **\[安全红线\] Query Scope / Consent Boundary (授权边界)**：此接口默认**仅针对公开索引层（Public Root Index）和公开广播凭证**进行推理，绝不触及/读取用户本地未公开私有数据。

## **🌐 Phase 6：万能解析与公共 AI 网关 (Universal Resolver & AI Endpoint)**

**状态**：📋 概念设计锁定

*战略纠偏：明确区分静态解析与动态查询，绝不混淆。*

* **静态 Resolver 模式 (默认纯前端)**：部署于 verihash.org (Pages托管)。不设数据库，纯 JS 拉取公开 Root Index 动态渲染卡片。  
* **Query Gateway 模式 (API 路由)**：提供类似 /?query=xxx 的 AI 提问能力，该节点仅作反向代理，将请求转发回用户的 Phase 5B 公开查询服务。

## **🔐 Phase 7：本地容灾与加密冷备份 (Disaster Recovery & Cold Backup)**

**状态**：🔥 关键底座补回 (优先级极高)

*战略定位：消除用户“本地数据库损坏”、“换机丢失账本”的资产焦虑。广播是被看到，备份是别丢命。*

* **BYOS 加密冷备份 (Bring Your Own Storage)**：本地账本及底层凭证通过密码学加密打包。  
* **.vhb 格式定义 (VeriHash Backup)**：标准化的加密导出压缩包。  
* **助记词恢复**：新设备只需引入 .vhb 文件并输入助记词，瞬间解密并重建所有本地账本和哈希链，保证工作量资产永续存在。

## **🗃️ Phase 8：沉浸式账本与状态层 (The Immutable Ledger & Status)**

**状态**：✅ 核心已完成 / 迭代中

* **广播状态机 (Broadcast Status Layer)**：精确的异步追踪机：pending, success, failed, last\_error。解耦本地存储与网络波动。
